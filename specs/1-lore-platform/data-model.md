# Data Model: Lore Platform

## Entity: Chunk

The fundamental unit of stored context. Represents a single piece
of organizational knowledge extracted from a source.

| Field          | Type           | Constraints                    |
|----------------|----------------|--------------------------------|
| id             | UUID           | PK, auto-generated             |
| content        | TEXT           | NOT NULL                       |
| embedding      | VECTOR(768)    | Nullable (populated at ingest) |
| content_type   | TEXT           | code, pull_request, adr, doc, spec, runbook |
| team           | TEXT           | Derived from CODEOWNERS        |
| repo           | TEXT           | Source repository               |
| file_path      | TEXT           | Source file path                |
| author         | TEXT           | Last author                    |
| ingested_at    | TIMESTAMPTZ    | Auto-set on insert             |
| metadata       | JSONB          | Type-specific fields (below)   |
| search_tsv     | TSVECTOR       | Generated from content (BM25)  |

**Indexes:**
- ScaNN index on `embedding` (vector cosine ops, `num_leaves=64`).
- GIN index on `search_tsv` (BM25 keyword search).

**Schema isolation:** Each team has an identical `chunks` table in
their own schema. MCP server for team X queries schema X +
`org_shared`.

### Content-Type Specific Metadata (JSONB)

**code:**
```json
{
  "symbol_name": "ChargeBuilder",
  "dependencies": ["PaymentGateway", "IdempotencyKey"],
  "related_pr": "PR#1201",
  "related_adrs": ["ADR-042"]
}
```

**pull_request:**
```json
{
  "pr_number": 1201,
  "merged_at": "2025-06-15T14:30:00Z",
  "files_changed": ["src/charges/builder.ts"],
  "adr_refs": ["ADR-042"],
  "alternatives_rejected": "BigDecimal (cross-stack inconsistency)"
}
```

**adr:**
```json
{
  "adr_number": 42,
  "status": "accepted",
  "domains": ["payments", "billing"],
  "related_prs": ["PR#1201", "PR#1847"],
  "superseded_by": null
}
```

**doc:**
```json
{
  "space": "Engineering",
  "page_title": "Payment Processing Overview",
  "section": "Idempotency",
  "last_edited": "2025-09-01",
  "staleness_flag": false
}
```

**spec:**
```json
{
  "feature_name": "idempotency-v2",
  "content_subtype": "constitution",
  "linked_pr": "PR#2103"
}
```

**runbook:**
```json
{
  "severity": "P1",
  "trigger": "webhook_processing_error_rate > 5%",
  "last_incident": "2025-07-22",
  "staleness_flag": false
}
```

### Sensitivity Classification

| Level        | Behavior                                     |
|--------------|----------------------------------------------|
| (default)    | Searchable by team + org_shared queries       |
| restricted   | Excluded from general search. PII detected.  |
| internal     | Excluded from external contractor access.     |

Classification runs at ingest time. Chunks with email addresses or
card-like patterns are auto-classified as `restricted`.

### Lifecycle

- **Create:** Ingest pipeline creates chunks with embedding.
- **Update:** Nightly re-index upserts (match on `file_path` +
  `content_type` + `repo`).
- **Delete:** Hard-delete when source no longer exists. No
  soft-delete, no retention.

## Entity: Klaus Task

Represents a unit of work delegated to a Klaus cluster agent.

| Field       | Type         | Constraints                      |
|-------------|--------------|----------------------------------|
| task_id     | STRING       | PK, generated by Klaus           |
| task        | TEXT         | Natural language description     |
| context     | JSONB        | Context bundle (see below)       |
| priority    | ENUM         | low, normal, high                |
| status      | ENUM         | submitted, running, completed, failed, timed_out |
| submitted_at | TIMESTAMPTZ | Auto-set                        |
| completed_at | TIMESTAMPTZ | Set on completion/failure       |
| output      | TEXT         | Result text (on completion)      |
| failure_reason | TEXT      | Error details (on failure)       |

**Context bundle structure:**
```json
{
  "beads_task_id": "bd-a3f8",
  "spec_content": "...",
  "constitution_content": "...",
  "seed_chunks": ["...", "..."],
  "branch": "feature/idempotency-v2"
}
```

**Lifecycle:**
```
submitted → running → completed (output stored)
                   → failed (reason stored, Beads claim released)
                   → timed_out (treated as failed)
```

No automatic retry. Developer decides whether to resubmit.

## Entity: Beads Task

Managed by the `bd` CLI, stored in Dolt. Listed here for reference
since the Lore platform depends on its structure.

| Field       | Type         | Constraints                      |
|-------------|--------------|----------------------------------|
| id          | STRING       | PK, generated by bd              |
| title       | TEXT         | Task description                 |
| status      | ENUM         | pending, in_progress, done       |
| claimed_by  | TEXT         | Agent or developer identifier    |
| version     | INTEGER      | Optimistic locking counter       |
| parent_id   | STRING       | FK to parent task (epics)        |
| created_at  | TIMESTAMPTZ  | Auto-set                        |
| updated_at  | TIMESTAMPTZ  | Updated on every mutation        |

**Dependencies:** Stored as edges in a separate table.

| Field      | Type   | Constraints                        |
|------------|--------|------------------------------------|
| child_id   | STRING | FK to task (blocked task)          |
| parent_id  | STRING | FK to task (blocking task)         |

**Optimistic locking:**
- Every mutation increments `version`.
- `bd update --claim` reads current version, writes `version + 1`.
- If version changed between read and write, operation is rejected
  with a clear error.

## Entity: Langfuse Trace

Stored in Langfuse's Cloud SQL database, exported to BigQuery.

| Field        | Type        | Notes                            |
|--------------|-------------|----------------------------------|
| trace_id     | STRING      | Generated by Langfuse            |
| name         | STRING      | `context-retrieval`              |
| namespace    | STRING      | Team schema name                 |
| query        | TEXT        | Search query                     |
| top_score    | FLOAT       | Highest RRF score                |
| result_count | INTEGER     | Number of results returned       |
| tags         | STRING[]    | `low-confidence` if score < 0.72 |
| gap_candidate | BOOLEAN    | True if low-confidence           |
| timestamp    | TIMESTAMPTZ | Auto-set                        |

**Gap detection query (BigQuery):**
Traces where `gap_candidate = true`, grouped by embedding similarity
over a 7-day window, clusters with 3+ occurrences flagged.

## Entity Relationships

```
Chunk ──(content_type: pull_request)──> metadata.adr_refs ──> Chunk (content_type: adr)
Chunk ──(content_type: code)──> metadata.related_pr ──> Chunk (content_type: pull_request)
Chunk ──(content_type: adr)──> metadata.superseded_by ──> Chunk (content_type: adr)
Chunk ──(content_type: spec)──> metadata.linked_pr ──> Chunk (content_type: pull_request)

Klaus Task ──> context.beads_task_id ──> Beads Task
Klaus Task ──> context.seed_chunks ──> Chunk (pre-fetched)

Langfuse Trace ──> namespace ──> Team Schema ──> Chunk (queried)
```

In Phase 3, these implicit relationships become explicit edges in
the GraphRAG knowledge graph, enabling multi-hop traversal queries.
