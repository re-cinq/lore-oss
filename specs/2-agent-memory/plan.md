## Implementation Phases

### Phase 1: Memory MCP Tools (1 week)

#### 1.1 Agent ID Generation

**Files:** `mcp-server/src/agent-id.ts`, `scripts/install.sh`

`install.sh` gains a new step between step 8 (init beads) and step 9 (AgentDB):

```bash
# --- Generate agent ID ---
generate_agent_id() {
  CURRENT_STEP="generate agent ID"
  AGENT_ID_FILE="$HOME/.lore/agent-id"
  mkdir -p "$HOME/.lore"
  if [ ! -f "$AGENT_ID_FILE" ]; then
    uuidgen > "$AGENT_ID_FILE"
    echo "[lore] Agent ID generated: $(cat "$AGENT_ID_FILE")"
  else
    echo "[lore] Agent ID exists: $(cat "$AGENT_ID_FILE")"
  fi
}
```

`agent-id.ts` resolves the agent ID at runtime:

1. If `agent_id` parameter is explicitly passed to any tool call, use it.
2. Else if `LORE_AGENT_ID` env var is set (Klaus pods set this to pod name), use it.
3. Else read `~/.lore/agent-id`.
4. Else generate a UUID, write it to `~/.lore/agent-id`, return it.

**Verification:** `install.sh` creates the file. MCP server reads it on first memory tool call. Klaus pods use pod name via env var.

#### 1.2 PostgreSQL Schema

**File:** `specs/2-agent-memory/contracts/db-schema.md` (DDL), applied via `scripts/infra/setup-memory-schema.sh`

All tables in a `memory` schema within the existing `lore` database on `lore-db-1`.

**`memory.memories`**

| Column       | Type           | Constraints                     |
|--------------|----------------|---------------------------------|
| id           | UUID           | PK, `gen_random_uuid()`         |
| agent_id     | TEXT           | NOT NULL, indexed               |
| key          | TEXT           | NOT NULL                        |
| value        | TEXT           | NOT NULL                        |
| embedding    | VECTOR(768)    | Nullable (populated on write)   |
| version      | INTEGER        | NOT NULL, starts at 1           |
| is_deleted   | BOOLEAN        | DEFAULT FALSE                   |
| ttl_seconds  | INTEGER        | Nullable                        |
| expires_at   | TIMESTAMPTZ    | Nullable, computed from ttl     |
| created_at   | TIMESTAMPTZ    | DEFAULT now()                   |
| pool_id      | UUID           | Nullable FK to shared_pools     |

- UNIQUE constraint on `(agent_id, key, version)`.
- HNSW index on `embedding` for vector search.
- Partial index: `WHERE is_deleted = FALSE AND (expires_at IS NULL OR expires_at > now())` for active memory queries.
- The `version` column is the latest version for this key. Previous versions live in `memory_versions`.

**`memory.memory_versions`**

| Column       | Type           | Constraints                     |
|--------------|----------------|---------------------------------|
| id           | UUID           | PK, `gen_random_uuid()`         |
| memory_id    | UUID           | FK to memories.id               |
| version      | INTEGER        | NOT NULL                        |
| value        | TEXT           | NOT NULL                        |
| embedding    | VECTOR(768)    | Nullable                        |
| created_at   | TIMESTAMPTZ    | DEFAULT now()                   |

- UNIQUE on `(memory_id, version)`.
- Every write (including the first) creates a version row. The `memories` table always holds the latest state.

**`memory.facts`**

| Column       | Type           | Constraints                     |
|--------------|----------------|---------------------------------|
| id           | UUID           | PK, `gen_random_uuid()`         |
| memory_id    | UUID           | FK to memories.id               |
| fact_text    | TEXT           | NOT NULL                        |
| embedding    | VECTOR(768)    | Nullable                        |
| created_at   | TIMESTAMPTZ    | DEFAULT now()                   |

- HNSW index on `embedding`.
- Facts are independently searchable. Linked back to parent memory.

**`memory.snapshots`**

| Column       | Type           | Constraints                     |
|--------------|----------------|---------------------------------|
| id           | UUID           | PK, `gen_random_uuid()`         |
| agent_id     | TEXT           | NOT NULL, indexed               |
| memory_refs  | JSONB          | NOT NULL (array of `{memory_id, version}`) |
| trigger      | TEXT           | 'manual' or 'auto'             |
| created_at   | TIMESTAMPTZ    | DEFAULT now()                   |

- Reference-based: stores memory IDs and version numbers, not full copies.
- Restore sets the `version` pointer on each memory to the snapshotted version.

**`memory.shared_pools`**

| Column       | Type           | Constraints                     |
|--------------|----------------|---------------------------------|
| id           | UUID           | PK, `gen_random_uuid()`         |
| name         | TEXT           | UNIQUE, NOT NULL                |
| created_by   | TEXT           | NOT NULL (agent_id)             |
| created_at   | TIMESTAMPTZ    | DEFAULT now()                   |

- Pools are created implicitly on first `shared_write`. The `memories` table references pools via `pool_id`.

**`memory.audit_log`**

| Column       | Type           | Constraints                     |
|--------------|----------------|---------------------------------|
| id           | UUID           | PK, `gen_random_uuid()`         |
| agent_id     | TEXT           | NOT NULL                        |
| operation    | TEXT           | NOT NULL (write, read, search, delete, snapshot, restore, shared_write, shared_read) |
| memory_key   | TEXT           | Nullable                        |
| pool_name    | TEXT           | Nullable                        |
| metadata     | JSONB          | Nullable (search query, result count, etc.) |
| created_at   | TIMESTAMPTZ    | DEFAULT now()                   |

- Append-only. No updates, no deletes.
- Indexed on `(agent_id, created_at)` for per-agent audit queries.

#### 1.3 MCP Memory Tools

**Files:** `mcp-server/src/memory.ts`, `mcp-server/src/index.ts`

Five tools registered in `index.ts` via `server.tool()`:

**`write_memory`**

```typescript
{
  key: z.string().describe('Memory key (e.g., "user-preference", "last-gap-run")'),
  value: z.string().describe('Memory value (text, any length)'),
  agent_id: z.string().optional().describe('Override agent ID. Defaults to ~/.lore/agent-id.'),
  ttl: z.number().optional().describe('Time-to-live in seconds. Omit for permanent.'),
  extract_facts: z.boolean().optional().describe('Extract individual facts from value (async).')
}
```

Behavior:
1. Resolve agent ID via `agent-id.ts`.
2. Generate embedding via existing `getQueryEmbedding()` in `db.ts`.
3. If key exists for this agent: increment version, insert into `memory_versions`, update `memories` row.
4. If key is new: insert into `memories` with version 1, insert into `memory_versions`.
5. If `ttl` provided: compute `expires_at = now() + interval '${ttl} seconds'`.
6. If `extract_facts = true`: queue async fact extraction (Phase 2 -- in Phase 1, log a warning and skip).
7. Log to `audit_log`.
8. Return: `{ key, version, agent_id, created_at }`.

**`read_memory`**

```typescript
{
  key: z.string().describe('Memory key to read.'),
  agent_id: z.string().optional(),
  version: z.string().optional().describe('"all" for full history, or specific version number.')
}
```

Behavior:
1. If `version = "all"`: return all versions from `memory_versions` ordered by version desc.
2. If `version` is a number: return that specific version.
3. Default: return latest from `memories` where `is_deleted = FALSE`.
4. Log to `audit_log`.

**`delete_memory`**

```typescript
{
  key: z.string().describe('Memory key to delete.'),
  agent_id: z.string().optional()
}
```

Behavior:
1. Set `is_deleted = TRUE` on the `memories` row. Does not delete `memory_versions`.
2. Log to `audit_log`.
3. Return confirmation with key and agent_id.

**`list_memories`**

```typescript
{
  agent_id: z.string().optional(),
  limit: z.number().default(50).describe('Max results.'),
  offset: z.number().default(0).describe('Pagination offset.')
}
```

Behavior:
1. Query `memories` where `is_deleted = FALSE AND (expires_at IS NULL OR expires_at > now())`.
2. Return: array of `{ key, version, created_at, has_facts, ttl_seconds }`.
3. Log to `audit_log`.

**`search_memory`**

```typescript
{
  query: z.string().describe('Natural language search query.'),
  agent_id: z.string().optional().describe('Scope to agent. Omit for cross-agent.'),
  pool: z.string().optional().describe('Search within a shared pool.'),
  limit: z.number().default(10)
}
```

Behavior:
1. Generate query embedding via `getQueryEmbedding()`.
2. Search both `memories.embedding` and `facts.embedding` using HNSW cosine distance.
3. Filter by agent_id if provided, or by pool if provided.
4. Exclude `is_deleted = TRUE` and expired memories.
5. Merge results by Reciprocal Rank Fusion (same pattern as existing `hybridSearch` in `db.ts`).
6. Return: array of `{ key, value, score, agent_id, source }` where source is "memory" or "fact".
7. Log to `audit_log` with query and result count.

#### 1.4 File-Backed Fallback

**File:** `mcp-server/src/memory-file.ts`

When PostgreSQL is unavailable (`isAlloyDbAvailable() = false`), all memory tools fall back to JSON files in `~/.lore/memory/`:

```
~/.lore/memory/
  <agent-id>/
    memories.json       # { key: { value, version, created_at, ttl_seconds, is_deleted } }
    versions.json       # { key: [{ version, value, created_at }] }
    snapshots.json      # [{ id, memory_refs, trigger, created_at }]
  audit.jsonl           # append-only, one JSON object per line
```

No embeddings in file mode -- `search_memory` degrades to case-insensitive substring matching (same pattern as Phase 0 `search_context` fallback). Fact extraction is skipped in file mode.

The fallback is transparent to the caller. Each tool handler checks `isAlloyDbAvailable()` first, same pattern as the existing `get_context` / `search_context` tools.

**Verification:** Unset `LORE_DB_HOST`, call `write_memory` + `read_memory` + `search_memory`. All work against local files.

#### 1.5 Phase 1 Verification

- `write_memory(key: "test", value: "hello world")` returns version 1 with agent ID.
- `read_memory(key: "test")` returns "hello world".
- `write_memory(key: "test", value: "updated")` returns version 2.
- `read_memory(key: "test", version: "all")` returns both versions.
- `search_memory(query: "greeting")` returns the "hello world" / "updated" memory by semantic similarity.
- `delete_memory(key: "test")` soft-deletes. Subsequent `read_memory` returns not found.
- `list_memories()` excludes deleted and expired entries.
- All of the above work in file-backed mode (no PostgreSQL).
- `install.sh` creates `~/.lore/agent-id` on fresh install.
- `lore-doctor.sh` checks agent ID file exists.

### Phase 2: Advanced Features (1 week)

#### 2.1 Fact Extraction

**File:** `mcp-server/src/facts.ts`

Async fact extraction triggered by `write_memory(..., extract_facts: true)`:

1. `write_memory` returns immediately after storing the raw value.
2. A background function calls the configured LLM to extract facts.
3. LLM configuration via env vars:
   - `LORE_FACT_LLM=claude` (default) / `openai` / `ollama`
   - `LORE_FACT_MODEL=claude-sonnet-4-20250514` (or equivalent)
   - `LORE_OLLAMA_URL=http://localhost:11434` (for local Ollama)
4. Prompt: "Extract individual factual statements from the following text. Return one fact per line. Be specific and self-contained -- each fact should be understandable without the others."
5. Each extracted fact is stored in `memory.facts` with its own embedding.
6. If the LLM is unreachable: log a warning, queue for retry. The memory is searchable as raw text until extraction completes. Retry via a simple in-process queue with exponential backoff (3 attempts, then give up and log).

**Auth pattern:** Claude API via `ANTHROPIC_API_KEY` env var (already present on Klaus pods). OpenAI same pattern. Ollama has no auth. Local dev uses whichever LLM is configured -- Ollama preferred for zero-credential compliance.

#### 2.2 Shared Memory Pools

**Tools:** `shared_write`, `shared_read` (registered in `index.ts`)

**`shared_write`**

```typescript
{
  pool: z.string().describe('Named pool (created implicitly on first write).'),
  key: z.string(),
  value: z.string(),
  agent_id: z.string().optional()
}
```

Behavior:
1. If pool does not exist in `shared_pools`: create it, set `created_by`.
2. Write to `memories` with `pool_id` set. Agent ID is attributed.
3. Embedding generated on write (same as private memories).
4. Log to `audit_log` with `pool_name`.

**`shared_read`**

```typescript
{
  pool: z.string(),
  key: z.string().optional().describe('Specific key, or omit to list all pool entries.')
}
```

Behavior:
1. If `key` provided: return that entry from the pool.
2. If no `key`: return all entries in the pool, paginated.
3. `search_memory(query, pool: "pool-name")` already handles pool-scoped search from Phase 1.

**File-backed fallback:** Shared pools stored in `~/.lore/memory/shared/<pool-name>/memories.json`. Works for single-machine testing. Cross-machine sharing requires PostgreSQL.

#### 2.3 Snapshots

**Tools:** `create_snapshot`, `restore_snapshot` (registered in `index.ts`)

**`create_snapshot`**

```typescript
{
  agent_id: z.string().optional(),
  trigger: z.enum(['manual', 'auto']).default('manual')
}
```

Behavior:
1. Query all active memories for the agent (not deleted, not expired).
2. Build `memory_refs`: array of `{ memory_id, version }` for each memory.
3. Insert into `snapshots`. No data copied -- references only.
4. Return: `{ snapshot_id, memory_count, created_at }`.

**`restore_snapshot`**

```typescript
{
  snapshot_id: z.string()
}
```

Behavior:
1. Read `memory_refs` from the snapshot.
2. For each ref: if the memory's current version is higher than the snapshotted version, revert `memories.version` and `memories.value` to the snapshotted version (read from `memory_versions`).
3. Memories not in the snapshot but created after it: soft-delete them.
4. Return: `{ restored_count, snapshot_created_at }`.
5. Target: under 1 second for 10,000 memories. The operation is a single transaction with bulk UPDATE from a JOIN on `memory_versions`.

#### 2.4 TTL and Expiration

**File:** `k8s/memory-ttl-cronjob.yaml`

CronJob in `klaus` namespace, runs hourly:

```yaml
schedule: "0 * * * *"
```

Job: connect to PostgreSQL, run:

```sql
DELETE FROM memory.memories
WHERE expires_at IS NOT NULL AND expires_at < now() - interval '24 hours';

DELETE FROM memory.memory_versions
WHERE memory_id NOT IN (SELECT id FROM memory.memories);

DELETE FROM memory.facts
WHERE memory_id NOT IN (SELECT id FROM memory.memories);
```

Expired memories are excluded from search immediately (via the partial index WHERE clause). The CronJob does the hard delete 24 hours after expiration as a cleanup pass.

#### 2.5 Agent Health and Stats Tools

**Tools:** `agent_health`, `agent_stats` (registered in `index.ts`)

**`agent_health`**

```typescript
{ agent_id: z.string().optional() }
```

Returns: `{ agent_id, memory_count, last_active, snapshot_count, oldest_memory, newest_memory }`.

**`agent_stats`**

```typescript
{ agent_id: z.string().optional() }
```

Returns: `{ agent_id, total_memories, total_facts, total_searches, shared_pools, memories_by_day (last 7 days) }`. Aggregated from `audit_log`.

### Phase 3: Web UI (2 weeks)

#### 3.1 Technology Choice

**Framework:** Next.js (App Router) with server components. Deployed on GKE in the `lore-ui` namespace.

Rationale:
- Server components query PostgreSQL directly -- no separate API layer needed.
- App Router provides file-based routing matching the page structure.
- Lightweight: no state management library, no GraphQL. Just SQL queries rendered as HTML.
- Same pattern as other internal tools on the GKE cluster.

**Auth:** Google Workspace OIDC via NextAuth.js. Same Google Workspace identity used across all GKE services. Only `@re-cinq.com` domain allowed.

**Database access:** Direct SQL queries against the same PostgreSQL instance (`lore-db-1`). Read-only connection. The UI uses a dedicated `lore_ui` database user with SELECT-only grants on `memory.*` and `org_shared.*`.

#### 3.2 Pages

**Agent Overview (`/`)**

- Lists all known agents (distinct `agent_id` values from `memories` + `audit_log`).
- Per agent: memory count, last active time, snapshot count, fact count.
- Sortable by last active, memory count.
- Link to drill into each agent.

**Memory Browser (`/agents/[id]`)**

- Lists all memories for an agent, paginated.
- Each memory shows: key, value (truncated), version, created_at, has_facts, ttl.
- Expand to see full value + version history + extracted facts.
- Soft-delete button (for platform engineers).

**Search (`/search`)**

- Full-text + semantic search across all agent memories.
- Scope filter: specific agent, specific pool, or all.
- Results show: key, value excerpt, score, agent_id, source (memory/fact).
- Uses the same hybrid search query as the MCP `search_memory` tool.

**Audit Trail (`/audit`)**

- Filterable log of all memory operations.
- Filters: agent_id, operation type, date range.
- Shows: timestamp, agent_id, operation, key, metadata.
- Paginated, newest first.

**Shared Pools (`/pools`)**

- Lists all shared pools with entry counts and contributing agents.
- Drill into a pool to browse entries.
- Search within a pool.

**Non-Developer Features (`/tasks`, `/specs`, `/gaps`)**

For product owners and managers who do not use Claude Code:

- `/tasks`: View current Beads tasks across agents. Add new tasks (writes to Beads via API).
- `/specs`: Browse `.specify/` specs ingested into PostgreSQL. Add new spec text (writes to `org_shared.chunks` with `content_type = 'spec'`).
- `/gaps`: Review gap detection drafts (PRs labelled `context-gap-draft`). Approve/reject directly from the UI (triggers GitHub API via server action).

## Risk Register

| Risk | Impact | Likelihood | Mitigation |
|------|--------|------------|------------|
| Embedding latency on write (Vertex AI round-trip) | Medium | Medium | Async embedding -- write succeeds immediately with null embedding, background job fills it. Search degrades to keyword-only until embedding arrives. |
| Fact extraction LLM unavailable | Low | Medium | Write succeeds without facts. Retry queue with 3 attempts + exponential backoff. Memory searchable as raw text. |
| File-backed fallback feature drift | Medium | Medium | Shared test suite that runs against both backends. CI runs tests in file-backed mode. |
| Schema migration on existing CNPG instance | Medium | Low | Memory tables in dedicated `memory` schema -- no risk to existing `org_shared` / team schemas. Migration script is additive only (CREATE, never ALTER existing tables). |
| Web UI scope creep (feature requests from non-developers) | High | High | Strict scope: read-only views + task/spec input. No workflow automation, no dashboards, no notifications. Revisit after 1 month of usage. |
| Snapshot restore correctness with concurrent writes | Medium | Low | Restore runs in a serializable transaction. Concurrent writes during restore are blocked briefly. Acceptable for sub-second restore. |
| Agent ID collision across machines | Low | Low | UUID v4 has negligible collision probability. If it ever happens, agent memories merge -- no data loss, just mixed context. |

## Critical Path

```
Agent ID generation (Phase 1, day 1)
  -> memory schema DDL (Phase 1, day 1)
    -> write_memory + read_memory with PostgreSQL (Phase 1, day 2)
      -> search_memory with embeddings (Phase 1, day 3)
        -> file-backed fallback (Phase 1, day 4)
          -> fact extraction (Phase 2, day 1)
            -> shared pools (Phase 2, day 2)
              -> snapshots (Phase 2, day 3)
                -> Web UI scaffold + auth (Phase 3, week 1)
                  -> all UI pages (Phase 3, week 2)
```

The critical dependency is embedding on write. If Vertex AI embedding latency is too high for synchronous writes (>50ms target per NFR-1), switch to async embedding on day 3 rather than day 4. This does not block any downstream work -- search falls back to keyword matching for un-embedded memories.

The file-backed fallback must be built in Phase 1 (not deferred) because it is a P1 constitution requirement. Without it, memory tools are unusable locally.
