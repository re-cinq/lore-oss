# Feature Specification: Auto-Episode Hooks

| Field          | Value                                       |
|----------------|---------------------------------------------|
| Feature        | Auto-Episode Hooks                          |
| Status         | Draft                                       |
| Created        | 2026-04-03                                  |
| Owner          | Platform Engineering                        |
| Priority       | P1 — Lower effort, high impact              |
| Motivation     | [Zep competitive research](../zep-competitive-research.md) |
| Depends on     | [Episode ingestion](../episode-ingestion/spec.md) (shipped) |

## Problem Statement

`write_episode` exists but agents must call it explicitly. In
practice, no agent remembers to do this — the most valuable
context (session summaries, debugging observations, PR review
feedback) is lost because nobody calls the tool.

Zep's insight was that ingestion should be passive. The system
captures everything; the agent doesn't decide what's worth
remembering. We have the ingestion pipeline but not the
automatic triggers.

## Vision

Claude Code hooks automatically call `write_episode` at key
moments — session end, after tool use, on PR review events.
Developers don't configure anything; the install script sets
it up. Every session leaves a trace that enriches the org's
knowledge graph and fact store.

## User Scenarios & Acceptance Criteria

### Scenario 1: Session Summary on Exit

**Actor:** Developer using Claude Code

**Flow:**
1. Developer works in Claude Code for 30 minutes.
2. Session ends (Ctrl+C, `/exit`, or timeout).
3. A hook fires that calls `write_episode` with a summary of
   the session: files changed, decisions made, problems hit.
4. Facts and graph entities are extracted automatically.
5. Next session, `search_memory` returns insights from this one.

**Acceptance Criteria:**
- Hook fires on session end without developer action.
- Episode content summarizes what happened (not a raw log dump).
- Summary is generated by a lightweight LLM call (Haiku).
- If the session made no meaningful changes (just a greeting),
  no episode is written.

### Scenario 2: PR Review Capture

**Actor:** Review agent or webhook handler

**Flow:**
1. A human posts a PR review with inline comments.
2. The review-reactor job (or webhook) captures the review text.
3. It calls `write_episode` with `source: "pr-review"` and
   `ref: "owner/repo#42"`.
4. Facts like "reviewer prefers explicit error types" are
   extracted and searchable org-wide.

**Acceptance Criteria:**
- PR review text is captured as an episode automatically.
- Reviewer preferences become searchable facts.
- No duplicate episodes for the same review.

### Scenario 3: CI Failure Capture

**Actor:** Nightly ingestion or webhook handler

**Flow:**
1. A CI run fails on a PR.
2. The failure log (last 500 lines) is captured as an episode
   with `source: "ci"` and `ref: "owner/repo#42"`.
3. Facts like "test X fails when Y" are extracted.
4. Next time an agent encounters a similar failure, it finds
   the relevant episode.

**Acceptance Criteria:**
- CI failures are captured with enough context to be useful.
- Log is truncated to avoid exceeding the 50K char limit.
- Duplicate failures (same error, same PR) are deduplicated
  via content hash.

## Functional Requirements

### FR-1: Session Summary Hook

- FR-1.1: Add a `PreExit` or `SessionEnd` hook to the Lore
  install script that calls `write_episode` via the MCP server.
- FR-1.2: The hook generates a summary by reading the
  conversation context (last N tool calls + responses) and
  calling a lightweight LLM (Haiku) to summarize.
- FR-1.3: Skip if the session had fewer than 3 tool calls
  (trivial session, not worth capturing).
- FR-1.4: Episode source is `"session"`, ref is the repo name.

### FR-2: PR Review Capture

- FR-2.1: Extend the `review-reactor` job to call `write_episode`
  when it detects new PR reviews.
- FR-2.2: Episode content includes: reviewer name, review body,
  inline comments with file paths and line numbers.
- FR-2.3: Source is `"pr-review"`, ref is `"owner/repo#PR"`.
- FR-2.4: Deduplicated by content hash (same review text = no-op).

### FR-3: CI Failure Capture

- FR-3.1: Add episode capture to the CI webhook handler or
  nightly job.
- FR-3.2: Capture the last 500 lines of the failed job log.
- FR-3.3: Source is `"ci"`, ref is `"owner/repo#PR"` or
  `"owner/repo@sha"`.

### FR-4: Install Script Integration

- FR-4.1: `install.sh` configures the session summary hook
  automatically.
- FR-4.2: No additional configuration needed by the developer.
- FR-4.3: The hook is idempotent — re-running install doesn't
  duplicate it.

## Non-Functional Requirements

### NFR-1: Performance

- Session summary hook adds < 5 seconds to session exit.
- Summary LLM call uses Haiku (fast, cheap).

### NFR-2: Cost

- Session summary: ~$0.001 per session (Haiku).
- PR review capture: no additional LLM cost (raw text, fact
  extraction is already async).

## Scope Boundaries

### In Scope

- Session summary hook in install script.
- PR review capture in review-reactor.
- CI failure capture.
- Install script integration.

### Out of Scope

- Real-time streaming of tool use events (too noisy).
- Capturing every conversation turn (too much data).
- Custom hook configuration per user (everyone gets the same).

## Success Criteria

1. After install, every Claude Code session > 3 tool calls
   automatically produces an episode without developer action.
2. PR review feedback is captured and searchable org-wide within
   30 seconds of the review being posted.
3. CI failure context is searchable by agents working on the
   same codebase.
4. Episode volume is manageable — no more than 1 episode per
   session, 1 per PR review, 1 per CI failure.
